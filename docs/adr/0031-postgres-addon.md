# ADR-0031: The Postgres add-on — one shared instance, a database and role per app

## Status

Accepted. The first **backend** building block, extending the add-on model
([ADR-0025](0025-building-block-addons.md)) from observability into the data tier. It builds on
the per-app secrets fabric ([ADR-0029](0029-secrets-through-the-control-plane.md)) — the generated
`DATABASE_URL` lands in the app's per-app Secret — and the Burrow-owned credential path
([ADR-0030](0030-credentials-through-the-control-plane.md)) for the instance's superuser password.

## Context

Everything shipped so far runs an app and the services *around* it (routing, TLS, logs, metrics,
cache). The first thing a real app needs is a **database** ([North Star](../ROADMAP.md): the agent
deploys *and operates the whole stack*, not just the container). Postgres is the near-universal
default. We want the same one-ask flow the rest of Burrow has: the agent says "give this app a
database," Burrow provisions one on the user's own cluster, wires it into the app, and operates it
— with no human handling a connection string and no secret crossing MCP.

Three forces shape the design:

1. **Isolation vs. resource cost.** A dedicated Postgres instance per app is the strongest
   isolation but the heaviest (every app = a Postgres pod + a volume). A single instance shared
   across the user's apps is far lighter and is what a small operator wants by default. The
   maintainer's call: **one shared instance now; per-app instances are a later opt-in a user will
   ask for when they have a real need** (a hard failure-domain or resource boundary).
2. **"Shared instance" must still mean real isolation between apps.** Sharing a Postgres *server*
   is fine; sharing a *database* is not — app A must never read app B's tables. Postgres gives us
   the clean boundary for free: a **database per app** owned by a **role per app**, with `CONNECT`
   revoked from `PUBLIC`. Same server, isolated data. This costs no extra pods or volumes — the
   expensive thing is instances, not databases.
3. **The Burrow/app boundary is non-negotiable.** This add-on instance is **never** burrowd's own
   control-plane Postgres ([ADR-0012](0012-in-cluster-postgres.md)). That instance holds Burrow's
   state and lives in the control-plane namespace; apps get no access to it, ever. The add-on
   Postgres is a separate instance in the add-on namespace (`burrow-addons`), and apps never share
   anything with Burrow's own database.

## Decision

**Install one Postgres instance as an add-on; on attach, give each app its own database and login
role inside it, and write the generated `DATABASE_URL` into the app's per-app Secret.**

### The instance — `addon install postgres`

- A catalog entry (`AddonPostgres`) deploys the official **`postgres`** image (PostgreSQL License —
  BSD-style, clears the Apache/MIT/BSD install bar of [ADR-0025](0025-building-block-addons.md))
  through the existing stateful add-on path: a `Recreate` Deployment + a Service +
  a PersistentVolumeClaim, in `burrow-addons`, reached at `burrow-postgres.burrow-addons.svc:5432`.
- **The superuser password is a Burrow-owned credential, generated by burrowd at install.** burrowd
  generates a strong random password and writes it to a Secret **`burrow-postgres`** in
  `burrow-addons`; the Postgres pod reads it as `POSTGRES_PASSWORD` via `valueFrom.secretKeyRef`
  (the superuser role is a fixed `burrow_admin`). The password is generated server-side, never
  returned by any API, never logged, never sent over MCP — the same discipline as
  [ADR-0029](0029-secrets-through-the-control-plane.md)/[0030](0030-credentials-through-the-control-plane.md).
  It lives in `burrow-addons` (not the control-plane `burrow-credentials` Secret) because a pod can
  only mount a Secret in its own namespace.

### Per-app provisioning — `addon attach postgres <app>`

burrowd, connecting to the instance as `burrow_admin`, runs idempotent admin SQL:

```sql
CREATE ROLE "app_<app>" LOGIN PASSWORD '<generated>';     -- one role per app
CREATE DATABASE "<app>" OWNER "app_<app>";                -- one database per app
REVOKE CONNECT ON DATABASE "<app>" FROM PUBLIC;           -- only this app's role may connect
GRANT  CONNECT ON DATABASE "<app>" TO "app_<app>";
```

Then it composes the connection string and writes it as the **`DATABASE_URL`** key of the app's
per-app Secret (`burrow-app-<app>-secrets`) via the [ADR-0029](0029-secrets-through-the-control-plane.md)
`SetSecretValue` path, and restarts the app so `envFrom` picks it up:

```
DATABASE_URL = postgres://app_<app>:<generated>@burrow-postgres.burrow-addons.svc:5432/<app>?sslmode=disable
```

(`sslmode=disable` is the in-cluster default — traffic stays inside the cluster network; TLS to
the database is a later hardening, like ingress TLS was for the app edge.)

- **App and database names are validated against a strict `[a-z][a-z0-9-]*` identifier pattern
  before they reach SQL, and every identifier is quoted.** App names already match this (they are
  Kubernetes resource names); the validation is defense-in-depth so a name can never carry SQL.
- **Re-attach is idempotent and rotates:** if the role/database already exist, burrowd issues
  `ALTER ROLE ... PASSWORD` with a fresh secret and rewrites `DATABASE_URL`. No error, no orphaned
  state.

### Why attach is safe to expose over MCP

The agent supplies only the **app name** and the add-on type — no secret crosses MCP. burrowd
**generates** the password and `DATABASE_URL` server-side and injects them into the app's Secret;
the value is never returned to the agent, never logged, never put in the database (only the *fact*
of attachment and the key name are recorded). So `addon attach postgres <app>` is an MCP tool — it
is exactly the North-Star flow: the agent asks for a database, Burrow provisions it and wires it
in, and the agent then writes integration code that reads `DATABASE_URL`. This does **not** weaken
[ADR-0004](0004-code-never-over-mcp.md)/[0029](0029-secrets-through-the-control-plane.md): those
forbid a secret *value* traveling over MCP; here no value does.

### Teardown

- **`addon detach postgres <app>`** removes the `DATABASE_URL` key and, behind a **confirm**
  guardrail (it destroys data), `DROP DATABASE`/`DROP ROLE` for that app.
- **`addon uninstall postgres`** removes the Deployment, Service, PVC, and the `burrow-postgres`
  Secret through the existing add-on removal path (also a confirm guardrail — it destroys *every*
  app's database).

### RBAC

burrowd's `burrow-addons` Role gains its first secrets grant — namespace-scoped `get`, `create`,
`update`, `delete` on `secrets` in `burrow-addons` — so it can create the `burrow-postgres`
superuser Secret and read it back to run admin SQL. This stays within the add-on namespace; it adds
nothing in the control-plane or app namespaces, and there is still **no `ClusterRole`**. burrowd
reuses its existing Postgres client (pgx, already a dependency for its own state store) to reach the
add-on instance — no new dependency.

## Consequences

- **The first one-ask database.** "Give app `web` a Postgres database" → an instance on the user's
  cluster, an isolated database + role, `DATABASE_URL` in the app's environment, the app restarted
  to see it — with no human touching a connection string.
- **Real isolation on a shared instance.** Database-per-app + role-per-app + `REVOKE CONNECT FROM
  PUBLIC` means app A's role cannot connect to app B's database. One pod, one volume, N isolated
  databases.
- **A clean upgrade path to per-app instances.** When a user needs a hard boundary, a future ADR
  adds `addon install postgres --dedicated <app>` (or similar) without changing the attach/secret
  contract — the app still just reads `DATABASE_URL`.
- **BYO reaches the app identically.** A user on Neon/Supabase sets `DATABASE_URL` with `app secret
  set`; a provisioned instance writes the same key. The app is oblivious to which it got — the
  North-Star "no lock-in" property.
- **New surface:** an `AddonPostgres` catalog entry; the admin-SQL provisioner behind a seam (so it
  fakes in unit tests); `addon attach`/`detach` commands and their MCP tools; the `burrow-addons`
  secrets grant; and a deterministic k3d e2e (install → attach → app reads `DATABASE_URL` and
  round-trips a row → detach).
- **Not yet:** automated backups, connection pooling (PgBouncer), read replicas, major-version
  upgrades, and TLS to the database. Each is a later slice; this ADR is the provisioning-and-wiring
  core. Backups in particular are called out as the next thing a real operator will want.

## Rejected alternatives

- **A dedicated Postgres instance per app, by default.** Strongest isolation, but a pod + a volume
  per app is too heavy for the small-operator default. Deferred to an explicit opt-in for users
  with a real failure-domain need — they will ask.
- **One shared database, a schema or just a role per app.** Rejected: roles are cluster-global and
  a shared database leaks too easily (a missed `REVOKE` exposes another app's tables). A database
  per app is the clean, default-safe boundary and costs no extra resources.
- **Provision through MCP with the agent supplying the password.** Rejected: that would put a secret
  value on the MCP path, violating [ADR-0029](0029-secrets-through-the-control-plane.md). burrowd
  generating the value server-side keeps attach both agent-drivable *and* secret-free over MCP.
- **Store `DATABASE_URL` in the control-plane database.** Rejected: connection strings are secrets;
  they belong only in the app's Kubernetes Secret, never in Postgres or an API response
  ([ADR-0029](0029-secrets-through-the-control-plane.md)). The registry records attachment and the
  key name, not the value.
- **An AGPL or proprietary Postgres distribution / operator (e.g. a bundled CloudNativePG with
  copyleft tooling).** Out of scope for *install* under the [ADR-0025](0025-building-block-addons.md)
  license bar; the plain PostgreSQL-licensed image is enough for the provisioning core. A heavier
  operator can be *connected* later if a user already runs one.
