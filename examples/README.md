# Burrow examples — operate a real cluster with your agent

Each example is a **real-life test of the Burrow loop**: a scenario is planted on a live
cluster, you point your agent at Burrow, and the agent diagnoses and fixes the problem by
driving Burrow — the thing Burrow exists for. Then a grader script tells you whether it
actually worked.

These are run **by hand**. Setup and grading cost nothing; only your own interactive agent
session (your `claude` or other agent CLI, on your own plan) spends tokens. This is the
companion to the headless, asserted [`test/agent/diagnose.sh`](../test/agent/diagnose.sh):
that one proves the loop in CI-style automation, these let you *drive it yourself* and feel
how an operator would.

## The blind-workspace design

Every example has two nested directories:

```
crashloop-after-deploy/
├── README.md      ← ANSWER KEY: the planted bug + expected fix (you read this; the agent does not)
├── setup.sh       ← plants the broken state on the cluster
├── verify.sh      ← grades the outcome
└── workspace/     ← you launch your agent IN HERE
    ├── TICKET.md  ← the only context the agent gets: the symptom, like a support ticket
    └── .mcp.json  ← generated by setup; wires the agent to the Burrow MCP server
```

The agent's working directory is `workspace/`, and the answer key lives one level up in the
example's own `README.md`. So when you launch the agent in `workspace/`, the bug and its fix
are not in front of it — it only has the ticket and the live cluster, exactly like a real
on-call engineer. The bug is in the *running* system, not in the files, so the agent has to
actually use Burrow (`burrow_status`, `burrow_logs`, …) to find it.

> The boundary is a convention, not a sandbox: an agent *can* read `../README.md` if it
> decides to. Don't point it at the parent directory, and the test stays honest.

## Prerequisites

- **docker**, **k3d**, **ko** (`brew install ko`, https://ko.build), **kubectl**, **go**,
  **jq** — the setup builds Burrow and stands up a disposable local cluster.
- **Your agent CLI** — e.g. `claude` (Claude Code). It auto-loads the `.mcp.json` that setup
  writes into `workspace/`, so the Burrow tools are available with no extra config.

## Run an example

```sh
cd examples/crashloop-after-deploy
bash setup.sh                     # builds Burrow, stands up k3d, plants the broken state
```

Then, in a new terminal, play the operator:

```sh
cd examples/crashloop-after-deploy/workspace
claude                            # your agent; hand it TICKET.md and let it drive Burrow
```

When the agent says it's done, grade it:

```sh
cd examples/crashloop-after-deploy
bash verify.sh
```

The first `setup.sh` builds the images and creates the cluster (a couple of minutes); the
cluster named `burrow-examples` is **reused** across examples, so later ones are fast. When
you're finished:

```sh
bash examples/lib/teardown.sh     # delete the cluster
```
